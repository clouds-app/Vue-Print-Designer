# Vue Print Designer → React 18 迁移可行性分析

> 文档版本：2026-06-15  
> 分析对象：`vue-print-designer` v1.6.99  
> 目标：将 UI 层从 Vue 3 迁移至 React 18，保留打印渲染引擎与 Web Component 对外 API

---

## 1. 项目现状概览

### 1.1 技术栈

| 类别 | 当前技术 | 说明 |
|------|----------|------|
| UI 框架 | Vue 3.5 | Composition API + `<script setup>` |
| 状态管理 | Pinia 2.x | 2 个 Store（designer / templates） |
| 构建工具 | Vite 6 | 双配置：应用模式 + Web Component 库模式 |
| 语言 | TypeScript 5.6 | `vue-tsc` 类型检查 |
| 样式 | Tailwind CSS 3.4 | 全局 `style.css`，框架无关 |
| 图标 | unplugin-icons (vue3) | Material Symbols |
| 代码编辑器 | @guolao/vue-monaco-editor | 属性面板 Code 编辑 |
| 条码/二维码 | jsbarcode / qrcode | 框架无关 |
| 国际化 | 自研轻量 i18n | 基于 Vue `ref`，6 种语言 |
| 发布形态 | Web Component `<print-designer>` | ES + UMD 双格式 NPM 包 |

### 1.2 代码规模（`src/` 目录统计）

| 指标 | 数量 |
|------|------|
| Vue 组件 | **50** 个，约 **25,398** 行 |
| TypeScript 文件 | **60** 个，约 **22,245** 行 |
| 合计 | 约 **47,643** 行 |
| 核心 Store 逻辑 | designer actions 约 **4,000+** 行，templates 约 **1,200** 行 |
| Web Component 封装 | `web-component.ts` 约 **1,600** 行（公共 API） |
| 超大组件 | `PrintDesigner.vue`、`Canvas.vue`、`TableElement.vue` 均 **1,500+** 行 |

### 1.3 架构分层

```
┌─────────────────────────────────────────────────────────┐
│  Web Component 层 (<print-designer>)                     │
│  - 属性/方法/事件对外 API                                 │
│  - Shadow DOM + Monaco 样式同步                          │
├─────────────────────────────────────────────────────────┤
│  UI 层（Vue 组件，需重写）                                │
│  - 布局：Header / Panels / Modals / Toolbar              │
│  - 画布：Canvas / Ruler / Minimap / 拖拽选中              │
│  - 元素：Text / Image / Table / Barcode / QR / Shape     │
│  - 属性面板：Property* 系列组件                           │
├─────────────────────────────────────────────────────────┤
│  状态层（Pinia → 需迁移）                                 │
│  - useDesignerStore / useTemplateStore                   │
│  - composables（useTheme / usePrintSettings 等）         │
├─────────────────────────────────────────────────────────┤
│  框架无关层（可直接复用或微调）                            │
│  - utils/print/* 自研渲染引擎（PDF/图片/HTML/分页）       │
│  - types / constants / elementFactory / variables      │
│  - locales 文案文件（zh/en/ja/ko/de/zh-Hant）            │
│  - crudConfig / entityConstraints / toast / confirm    │
└─────────────────────────────────────────────────────────┘
```

### 1.4 产品特性与约束

- **跨框架集成**：README 明确支持 Vue / React / Angular / 原生 HTML，当前通过 Web Component 实现。
- **NPM 发布**：包名 `vue-print-designer`，已有下游用户与 demo 项目（vue-designer-sample）。
- **无自动化测试**：`test/wc-test.html` 为手工集成页，迁移回归风险较高。
- **Shadow DOM**：设计器运行在 Shadow Root 内，Monaco 样式需动态同步到 Shadow。

## 2. 可行性结论

### 2.1 总体评估：**可行，但属于中大型重写项目**

| 维度 | 评级 | 说明 |
|------|------|------|
| 技术可行性 | ★★★★☆ | React 18 可覆盖全部 UI 能力；打印引擎可原样保留 |
| 工作量 | ★★★★★ | UI + 状态约 3 万行需重写或等价迁移 |
| 风险 | ★★★★☆ | 无单测、Canvas 交互复杂、WC+Shadow DOM 有坑 |
| 业务价值 | ★★★☆☆ | 取决于是否要统一 React 技术栈或独立 React 产品线 |
| 对外 API 兼容 | ★★★★☆ | Web Component 接口可保持不变，降低集成方成本 |

**结论**：迁移在工程上完全可行。最大成本在 **UI 组件重写** 与 **Pinia → React 状态方案**，而非打印核心逻辑。建议采用 **分阶段、可并行发布** 的策略，而非一次性 Big Bang 替换。

### 2.2 可复用 vs 需重写

#### 可直接复用（约 40%～45% 代码量）

| 模块 | 路径 | 复用方式 |
|------|------|----------|
| 打印渲染引擎 | `src/utils/print/**` | 原样保留，零框架依赖 |
| 类型定义 | `src/types/index.ts` | 原样或微调 |
| 工具函数 | `uuid` / `units` / `color` / `fonts` / `variables` / `os` | 原样保留 |
| 元素工厂 | `src/utils/elementFactory.ts` | 原样保留 |
| CRUD 配置 | `src/utils/crudConfig.ts` | 原样保留 |
| 实体权限 | `src/utils/entityConstraints.ts` | 原样保留 |
| 常量 | `src/constants/paper.ts` | 原样保留 |
| 国际化文案 | `src/locales/*.ts` | 保留 JSON 结构，换加载层 |
| SVG 模板 | `src/svg/templates.ts` | 原样保留 |
| Tailwind 样式 | `src/style.css` | 基本原样（去掉 Vue 特有选择器若有） |
| WC 类型声明 | `src/web-component.d.ts` | 作为兼容性契约保留 |

#### 必须重写（约 55%～60% 代码量）

| 模块 | 当前 | React 18 目标 |
|------|------|---------------|
| 全部 Vue 组件 | 50 个 `.vue` | 50+ 个 `.tsx` 组件 |
| Pinia Store | 2 Store + 8 action 模块 | Zustand / Jotai / Redux Toolkit |
| Composables | 6 个 | 自定义 Hooks |
| 自研 i18n | Vue `ref` + `install` | Context + `useSyncExternalStore` 或 react-i18next |
| Monaco 集成 | @guolao/vue-monaco-editor | @monaco-editor/react |
| 图标 | unplugin-icons/vue3 | unplugin-icons/react 或 lucide-react |
| 入口 | `main.ts` / `App.vue` | `main.tsx` / `App.tsx` |
| WC 挂载 | `createApp` + `h()` | `createRoot` from `react-dom/client` |
| 类型检查 | vue-tsc | tsc + @types/react |

### 2.3 主要风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| Canvas 拖拽/缩放/多选逻辑复杂 | 功能回归 | 先迁移 Store，再逐块迁移 Canvas；保留 wc-test 对照 |
| Shadow DOM + React 18 | 事件、Portal、样式隔离 | 模态框 Portal 目标改为 Shadow Root；Monaco 样式同步逻辑保留 |
| Pinia 响应式语义差异 | 细粒度更新行为变化 | 选用 Zustand + immer；对 `watch` 逻辑改为 `useEffect` + 选择器订阅 |
| provide/inject 依赖树 | 子组件获取上下文失败 | 建立 `DesignerContext`（root、modalContainer、instanceId 等） |
| NPM 包名与品牌 | 现有 Vue 用户混淆 | 新包 `react-print-designer`，WC 标签仍为 `<print-designer>`；文档分轨 |
| 无单测 | 回归成本高 | 迁移前先补 WC API 快照测试 + 核心 Store 单元测试 |
| 工期低估 | 项目延期 | 按模块估 8～14 人周（1 人全职），预留 30% 缓冲 |

## 3. 推荐目标技术栈（React 18）

```json
{
  "react": "^18.3.1",
  "react-dom": "^18.3.1",
  "zustand": "^5.x",
  "immer": "^10.x",
  "react-i18next": "^15.x",
  "@monaco-editor/react": "^4.x",
  "lodash": "^4.17.21",
  "jsbarcode": "^3.12.3",
  "qrcode": "^1.5.4",
  "tailwindcss": "^3.4.17",
  "vite": "^6.x",
  "@vitejs/plugin-react": "^4.x"
}
```

**选型理由简述：**

- **Zustand**：API 接近 Pinia，学习成本低；支持在 WC 外层创建独立 store 实例（多实例隔离）。
- **react-i18next**：6 语言已有结构化文案，迁移成本低于重写 i18n；也可延续自研 i18n + React Context。
- **@monaco-editor/react**：官方维护，Shadow DOM 下需同样处理 CSS 注入。
- **Strict Mode**：开发期开启，暴露副作用问题；Canvas 事件监听注意 cleanup。

### 阶段 2：状态层迁移（2～3 周）

**目标**：Pinia 逻辑等价迁移，UI 尚未接入也可独立验证。

**迁移顺序（由底向上）：**

1. `state.ts` → Zustand initial state
2. `getters.ts` → store selectors / derived hooks
3. `actions/history.ts` → 最小，验证 undo/redo
4. `actions/selection.ts` → 选中、框选
5. `actions/layer.ts` → 图层顺序
6. `actions/page.ts` → 页面增删
7. `actions/guide.ts` → 辅助线
8. `actions/table.ts` → 表格结构
9. `actions/element.ts` → 元素 CRUD、变换
10. `actions/core.ts` → 画布、缩放、网格
11. `stores/templates.ts` → 模板 CRUD + 本地/远程

**关键映射：**

| Vue / Pinia | React / Zustand |
|-------------|-----------------|
| `defineStore` | `create()` |
| `store.xxx = val` | `set(state => ...)` 或 immer |
| `computed(() => ...)` | `useDesignerStore(s => ...)` 选择器 |
| `watch(source, cb)` | `useEffect` + 订阅 / `useStore.subscribe` |
| `storeToRefs` | 直接 selector 取 primitive |
| `setActivePinia(pinia)` | 每 WC 实例 `createStore()` 工厂 |

**交付物**：Store 单元测试、与 Vue 版相同操作的 state 快照对比

## 6. 策略建议

### 6.1 推荐方案：Monorepo 双轨并行

```
vue-print-designer (现有)  ── 维护 bugfix，进入 maintenance
react-print-designer (新建) ── 主开发线，共享 packages/shared
```

**优点**：不阻断现有 Vue 用户；共享打印引擎避免双份维护；WC API 一致。  
**缺点**：短期内需维护两套 UI（或 Vue 进入 LTS 只修关键 bug）。

### 6.2 不推荐方案

| 方案 | 原因 |
|------|------|
| 用 React 包一层 Vue 微前端 | 体积翻倍，Shadow DOM 更复杂 |
| 仅改写 demo 不迁移 WC 核心 | 与 NPM 产品形态脱节 |
| 一次性全量切换无基线测试 | 回归不可控 |

### 6.3 若仅需在 React 项目中使用

**最低成本路径**：继续使用现有 `<print-designer>` Web Component，**无需迁移**。  
仅当以下需求成立时才值得全量迁移：

- 需要深度定制 UI 且 WC 扩展不足
- 团队统一 React 技术栈，不愿维护 Vue 构建链
- 需要 SSR / RSC（Web Component 不适合 SSR 场景）
- 要大幅重构架构（同步推进 React 重写）

## 附录 B：Vue → React 语法对照（高频）

| Vue 3 | React 18 |
|-------|----------|
| `<script setup>` + `defineProps` | `function Comp(props: Props)` |
| `defineEmits` | `props.onXxx` 或 callback props |
| `v-model="x"` | `value={x} onChange={setX}` |
| `v-if` / `v-for` | `{cond && ...}` / `map()` |
| `ref` + `.value` | `useState` / `useRef` |
| `computed` | `useMemo` |
| `watch` | `useEffect` |
| `onMounted` / `onUnmounted` | `useEffect(() => { return cleanup })` |
| `provide` / `inject` | `Context.Provider` + `useContext` |
