# Vue Print Designer → React 18 迁移执行文档

> 文档版本：v1.0  
> 文档日期：2026-06-15  
> 适用对象：AI 执行代理、开发人员、项目管理者  
> 目标：将现有 `vue-print-designer` 的 UI 层迁移至 React 18，同时保留打印渲染引擎、Web Component 对外 API 与核心业务行为。

---

## 1. 文档用途

本文档不是可行性分析，而是**迁移执行手册**。其作用是将迁移工作拆分为 AI 可逐步执行、可验证、可回滚的任务。

执行原则如下：

1. **先共享，后 UI**：优先迁移框架无关代码，再迁移状态层，最后迁移界面。
2. **先基线，后改造**：先冻结现有行为，再开始替换实现。
3. **先主链路，后增强**：优先完成核心编辑、画布、打印、导出和 Web Component 封装。
4. **单任务可验收**：每个任务必须有输入、输出、依赖、验收标准。
5. **避免大爆炸迁移**：禁止一次性重写所有模块后再统一验证。

---

## 2. 迁移目标

### 2.1 业务目标

- 将 UI 框架从 Vue 3 迁移到 React 18。
- 保持打印引擎和导出能力不变。
- 保持 `<print-designer>` Web Component 标签及其公共 API 兼容。
- 保持多语言、模板管理、元素编辑、打印预览、导出等核心能力。

### 2.2 技术目标

- 使用 React 18 + TypeScript 作为新 UI 运行时。
- 采用可隔离实例的状态管理方案，推荐 Zustand。
- 将框架无关逻辑抽离到共享层。
- 保持 Shadow DOM 运行模式。
- 对 Monaco、Tailwind、Portal、弹窗容器等做 Shadow DOM 兼容处理。

### 2.3 成功标准

- `wc-test.html` 的核心流程可通过回归验证。
- Web Component 方法、属性、事件保持兼容。
- 打印/导出/模板/编辑等关键路径行为一致或等价。
- React 版可作为独立产品线发布。

---

## 3. 执行总原则

### 3.1 任务拆分原则

每个任务都必须满足以下条件：

- 目标唯一。
- 输入明确。
- 输出明确。
- 依赖明确。
- 验收标准明确。
- 可独立验证。

### 3.2 推荐任务粒度

- 1 个任务 = 1 个可交付成果。
- 最佳工作量：0.5 到 2 天。
- 超过 3 天的任务必须拆分。

### 3.3 AI 执行规范

AI 在执行每一步时必须遵守：

1. 先读取任务目标与依赖。
2. 只改动当前任务所需文件。
3. 变更后立即做检查。
4. 若出现错误，优先修复当前任务范围内的问题。
5. 当前任务未验收前，不进入下一任务。

## 5. 任务执行顺序建议

推荐执行顺序如下：

```text
T0 基线建立
  ↓
T1 工程脚手架
  ↓
T2 共享逻辑迁移
  ↓
T3 状态层迁移
  ↓
T4 React 基础设施
  ↓
T5 UI 外壳迁移
  ↓

  ↓

  ↓
```

### 可并行的任务

- T2 与 T3 可以部分并行。
- T5 与 T7 可以并行推进。

### 强依赖任务

- 状态层未完成前，不建议开始画布交互迁移。
- 画布主链路未完成前，不建议封装最终 Web Component。
- Web Component 封装前必须先稳定核心行为。

## 7. 风险控制要求

### 7.1 必须控制的风险

- Canvas 拖拽、选区、缩放相关逻辑复杂。
- Shadow DOM 内样式和 Portal 容器处理复杂。
- TableElement 复杂度高，容易成为迁移瓶颈。
- Web Component 的实例隔离必须保持。
- 没有自动化测试时，回归风险较高。

### 7.2 风险控制手段

- 每个阶段结束后进行一次行为对照。
- 重要模块先迁逻辑层，再迁视图层。
- 保留 Vue 版基线作为对照。
- 提前补充关键单测和接口测试。
- 遇到复杂功能时优先拆分子任务。
